## Core principle
Flow is Atlas's living understanding of the project — not a diagram editor or planning board. The user explores, refines, and extends Atlas's understanding. They never recreate it manually.

**The sentence:** "Flow isn't a feature. It's the interface through which Atlas proves it's been paying attention."

## Nine principles (summary)
1. Atlas builds the graph — user explores it
2. Nodes show knowledge first, ask questions second
3. Chat + graph are one surface (bottom sheet, Apple Maps-style)
4. Designer/Builder/Storyteller = perspectives on one canonical graph, not separate datasets
5. Atlas owns canonical layout; users may temporarily reposition during a session
6. Aura layer order: graph → aura → bottom sheet (aura hides under the panel)
7. Never feels empty — nodes bloom from history; conversational empty state if none
8. Direction is subtraction — remove anything that feels like a planning board
9. Every tap answers a question: what do we know / why / where from / what's uncertain / what depends / what changed

## The guard rail
When new features are proposed (node editing, sticky notes, manual relationships, folders, colors) — evaluate here first. Those make sense in a graph editor. They do not make sense in Atlas's living memory.
