# Handoff — Tier 1 conversational extraction + gap surfacing

**Date:** 2026-07-08
**Repo:** `Axiom-Atlas` (Cloud Run backend)
**Owner:** run in Cursor against the backend repo
**Frontend companion:** already shipped — Tier 1 intake sheet no longer
auto-opens on project entry (`artifacts/atlas-frontend/src/pages/workspace.tsx`,
`src/pages/home.tsx`). See "Frontend state" below.

---

## Why

The 6-question Tier 1 intake sheet was slapping users the moment they
created a project. User feedback (locked 2026-07-08): "This is
bullshit and I need it to be invisible. It needs to come through
conversation." The sheet is preserved as a manual review surface
(`TIER1_INTAKE_OPEN_EVENT`) but must never auto-open again.

The backend now owns Tier 1: extract slot values from the actual chat
turns, mark which slots are still missing, and only surface an explicit
question when Atlas genuinely cannot infer an answer.

## Goal

Atlas earns Tier 1 answers from conversation. When something is truly
missing, the frontend surfaces **one** contextual inline question — not
a 6-step form.

Rule (product): **Atlas never asks a question it can reasonably infer the
answer to. When Atlas asks, it references what it already understands.**

---

## Tier 1 schema (frozen — matches frontend)

Six slots, all strings. See `artifacts/atlas-frontend/src/lib/tier1Memory.ts`
for canonical labels/hints.

| key             | Question shown to user                            |
| --------------- | ------------------------------------------------- |
| `building`      | What are you building?                            |
| `audience`      | Who is it for?                                    |
| `problem`       | What problem does it solve?                       |
| `outOfScope`    | What's explicitly out of scope?                   |
| `successSignal` | How will you know it's working?                   |
| `constraints`   | What constraints are you working within?          |

`Tier1Answers = { building, audience, problem, outOfScope, successSignal, constraints }`
— all strings, empty string = missing.

## Backend work

### 1. Extraction pass (piggyback on thinking-receipts)

There is already a fire-and-forget Haiku extraction pass after each
assistant turn that produces thinking-receipts. Extend that same pass to
also emit a Tier 1 slot update.

For each completed assistant turn on a project-scoped conversation:

1. Load current `tier1_memory` for `project_id`.
2. Call the extractor model with:
   - Rolling window of the last N turns (recommend 8 user+assistant
     turns; cap prompt at ~4k tokens).
   - Current known Tier 1 answers.
   - Instruction: "Return updated values ONLY for slots you can infer
     with high confidence from the transcript. Leave others as null.
     Do NOT invent. Do NOT overwrite an existing non-empty answer
     unless the user contradicted it in this turn."
3. Merge: for each slot the model returned non-null, write into
   `tier1_memory.answers`. Skip empty strings.
4. Persist. Emit whatever event / websocket the frontend already
   listens on for `TIER1_UPDATED_EVENT` (frontend fires this locally
   after writes; add server-side push if you have a channel, otherwise
   the frontend will refetch on its existing triggers).

**Confidence gate:** only merge slots with confidence ≥ 0.75. Below
that, treat as still missing — the user hasn't actually said it yet.

**Idempotency:** safe to re-run on the same turn; merging non-null
high-confidence values is stable.

**Model:** whatever Haiku-tier model the receipts pass already uses.
Do not add a new provider.

### 2. Gaps endpoint

Add:

```
GET /api/projects/:projectId/tier1-gaps
```

**Auth:** same session/bearer scheme as other project routes.
Reject if requester does not own the project.

**Response 200:**
```json
{
  "missing": ["audience", "successSignal"],
  "nextGap": {
    "key": "audience",
    "question": "Who is it for?",
    "hint": "Be specific. A named archetype beats a demographic.",
    "atlasContext": "You've said you're building a focus tool for solo operators. Who specifically — indie devs, writers, founders?"
  },
  "completeness": 0.66
}
```

Rules:
- `missing`: array of Tier1 keys where the stored answer is empty or
  never extracted with sufficient confidence.
- `nextGap`: the single most valuable slot to ask about next. Pick by
  fixed priority: `building` → `audience` → `problem` → `successSignal`
  → `outOfScope` → `constraints`. Return `null` when `missing` is empty.
- `atlasContext`: a **one-sentence** grounded reference to what Atlas
  already understands, generated by the same extractor. Must reference
  a real detail from the transcript. If none available, return `null`
  and the frontend will fall back to the plain `question`.
- `completeness`: filled slots / 6, 0..1.

**Response 404** if the project has no chat activity yet
(`nextGap` would be meaningless). Frontend treats 404 as "stay silent."

### 3. Existing endpoints (unchanged)

Keep `GET/POST/PUT /api/memory/tier1[/:projectId]` exactly as-is. The
review sheet (`Tier1IntakeSheet`) still uses them for manual editing.

### 4. Skip / opt-out

`tier1_memory.skippedAt` (already in schema per frontend types) should
suppress `nextGap` — return `nextGap: null` even when slots are missing.
Frontend already has "don't ask me" affordance in the sheet.

## Acceptance

- Create a new project. Type a build prompt. No sheet opens. Ever.
- After 2–3 conversational turns, `GET /api/memory/tier1/:id` shows
  slots the user actually addressed populated.
- `GET /api/projects/:id/tier1-gaps` returns the missing slots and a
  grounded `nextGap`.
- Manual sheet (via `TIER1_INTAKE_OPEN_EVENT`) still writes correctly.
- `skippedAt` suppresses `nextGap`.

## Non-goals

- No UI changes in this handoff (frontend piece is separate).
- No new AI provider; use the existing receipts extractor.
- No breaking changes to `/api/memory/tier1*`.
