# Handoff: Commit Carryover (ambient thread → committed project)

**Lane:** backend (Axiom-Atlas repo, Cloud Run) + DB (Supabase `osuasytymbzurjvklhde`).
**Frontend shell:** already wired in `src/components/workspace/ChatStream.tsx` (CommitThresholdMarker + CommitGreetingBubble) and `src/pages/workspace.tsx` (consumer at `useState` initializer + cleanup effect, lines ~3867).

Frontend reads two things on workspace mount when `?source=commit-carryover` is present:

1. `sessionStorage["atlas-commit-carryover-<projectId>"]` =
   ```json
   { "committedAt": "<ISO 8601>", "greeting": "<optional dynamic line from Atlas>" }
   ```
2. The carried-over chat thread (already loaded by the existing project messages fetch — no change needed there).

Frontend strips the `source` param + clears the storage key after consumption. Refresh-safe.

---

## What the backend must do

### 1. Persist commit metadata on the project row

Add two columns to `public.projects`:

| column            | type         | notes                                                        |
| ----------------- | ------------ | ------------------------------------------------------------ |
| `committed_at`    | `timestamptz`| Already present on some rows via `handle_new_user`; ensure it is set on every ambient→committed transition. |
| `commit_synthesis`| `text`       | 2–3 sentence recap of the shaping thread + one forward question. Generated by Atlas at commit time. |

(Run the migration in Cursor. Include `GRANT` + RLS review per project conventions.)

### 2. Generate the synthesis at commit time

When the user commits an ambient thread into a project (existing endpoint that creates/promotes the project from `/home`), call Atlas with the full thread and store the result in `projects.commit_synthesis`. Suggested system prompt shape (compose via `composeAtlasPrompt` per `atlas-core.ts` rule):

> "You just helped the user shape an ambient thought into a committed project. In 2–3 sentences, recap what they decided matters, what tension is still open, and end with ONE forward question. No greeting, no preamble, no 'welcome.' Write as continuation, not introduction."

### 3. Return the carryover payload on commit

The commit endpoint response should include:

```ts
{
  projectId: number;
  committedAt: string;     // ISO
  commitSynthesis: string; // greeting
}
```

### 4. Frontend hands off to workspace

The home-screen commit flow (already wired to redirect to the new project) needs to:

```ts
sessionStorage.setItem(
  `atlas-commit-carryover-${projectId}`,
  JSON.stringify({ committedAt, greeting: commitSynthesis }),
);
navigate(`/project/${projectId}?source=commit-carryover`);
```

This is the only frontend change still needed once the backend lands. The consuming component (`workspace.tsx`) already reads, renders, and clears.

---

## Behavior contract

- Marker reads "Committed as a project · {relative time}" in gold; tooltip shows the absolute timestamp.
- Greeting bubble renders **once**, at the tail of the carried thread, in Atlas's no-card style (per `header-layout` memory).
- If `greeting` is absent or empty, fallback copy: *"Okay — this is a project now. Where do you want to take it first?"*
- Workspace auto-scrolls to bottom on mount via the existing `[messages, chatPending]` effect — no extra plumbing required.
- Marker + greeting do not re-render on later sessions (storage key is cleared after first consumption; absence of `?source=commit-carryover` means no marker).
